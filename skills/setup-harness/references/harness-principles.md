# Harness Engineering Principles

> Distilled from "Harness engineering: leveraging Codex in an agent-first world" (OpenAI, Feb 2026).
> These principles govern every file generated by `/setup-harness`.

## The Ten Principles

### 1. CLAUDE.md Is a Table of Contents, Not an Encyclopedia

**Rule**: Keep CLAUDE.md concise (the article uses ~100 lines as a guideline). It points to deeper docs — it does not contain them.

**Why**: Context is a scarce resource. A giant instruction file crowds out the task, the code,
and relevant docs. The agent either misses key constraints or optimizes for wrong ones.
When everything is "important," nothing is.

**How**: CLAUDE.md contains: project name, language/framework, build/test/lint commands,
a table of pointers to `docs/`, a list of skills and agents, and 5 core principles. That's it.

### 2. Progressive Disclosure

**Rule**: Agents start with a small, stable entry point and are taught where to look next,
rather than being overwhelmed up front.

**How**: CLAUDE.md → ARCHITECTURE.md → domain-specific docs → design docs.
Each level is self-contained and links forward to the next level of detail.

### 3. Repository Is the System of Record

**Rule**: If knowledge isn't in the repo, it doesn't exist for the agent.

**Why**: From the agent's point of view, anything it can't access in-context while running
effectively doesn't exist. Knowledge in Google Docs, chat threads, or people's heads
is not accessible to the system.

**How**: All project knowledge lives in `docs/`. Design decisions are in `docs/design-docs/`.
Plans are in `docs/exec-plans/`. Architecture is in `ARCHITECTURE.md`.
Everything is versioned, co-located, and discoverable.

### 4. Agent Legibility Over Human Aesthetics

**Rule**: Optimize the repository for the agent's reasoning, not human readability preferences.

**Why**: The codebase is the agent's workspace. Making it navigable for the agent directly
increases output quality and reduces errors.

**How**: Clear file naming, consistent patterns, explicit module boundaries, structured
documentation with machine-parseable sections (tables, lists, headers).

### 5. Enforce Architecture Mechanically

**Rule**: By enforcing invariants — not micromanaging implementations — agents ship fast
without undermining the foundation.

**Why**: Documentation alone doesn't keep a codebase coherent. Rules in prose get ignored.
Rules in code get followed.

**How**: Architecture constraints go in `.claude/rules/architecture.md`. They are loaded
every session. Test conventions go in path-scoped rules that load only when editing tests.
Custom linters catch violations before they land.

### 6. Garbage Collection

**Rule**: Technical debt is a high-interest loan. Pay it down continuously in small
increments rather than letting it compound.

**Why**: Agents replicate patterns that already exist — even suboptimal ones. Without
continuous cleanup, drift becomes the default.

**How**: The `/harness-quality` skill grades each domain. The `@gardener` agent scans for
stale docs and opens fix-up tasks. The `/harness-sync` skill re-aligns docs with code.
Run these regularly, not on "cleanup Fridays."

### 7. Golden Principles in Code

**Rule**: Human taste is captured once, then enforced continuously on every line of code.

**Why**: In a human-first workflow, style rules feel pedantic. With agents, they become
multipliers: once encoded, they apply everywhere at once.

**How**: Golden principles live in `.claude/rules/`. They are specific, mechanical, and
verifiable. Not "write clean code" but "one concept per file, keep files focused" and
"parse external data at boundaries with schema validation."

### 8. Increasing Autonomy

**Rule**: Each capability unlocked enables the next. Build the stack bottom-up.

**Why**: Early progress is slower because the environment is underspecified. The agent
lacks tools, abstractions, and internal structure to make progress toward high-level goals.

**How**: Start with structure (CLAUDE.md, ARCHITECTURE.md). Add rules (boundaries,
conventions). Add skills (review, plan, quality). Add agents (reviewer, architect, gardener).
Each layer makes the next more effective.

### 9. Parse at Boundaries, Trust Internally

**Rule**: Validate ALL external data at system boundaries. Internal code trusts
already-validated data — no redundant checks.

**Why**: Agents are most effective in environments with strict boundaries and predictable
structure. Boundary validation creates that predictability.

**How**: Encode this in `.claude/rules/architecture.md`. External data (API responses,
user input, file reads, env vars) gets parsed/validated at entry. Everything downstream
works with typed, validated data.

### 10. Give a Map, Not a Manual

**Rule**: Give the agent a small map showing where to find things, not a 1,000-page
instruction manual that tries to cover everything.

**Why**: Too much guidance becomes non-guidance. A monolithic manual turns into a graveyard
of stale rules. It rots instantly. It's hard to verify.

**How**: CLAUDE.md is the map. `docs/` is the territory. Architecture docs, quality grades,
design docs, execution plans, product specs — each in its own file, each maintained
independently, each mechanically verifiable.

## The Layered Architecture Pattern

The article describes a layered architecture where dependencies flow in one direction.
The specific layers vary by project — they must be DISCOVERED from the actual codebase
(for existing projects) or RESEARCHED for the language/framework (for greenfield).

The universal rules are:
- Dependencies flow in ONE direction (no backward imports)
- Cross-cutting concerns enter through explicit interfaces (not scattered direct imports)
- Circular dependencies between domains are never allowed

The specific layer names, order, and structure are project-specific.
See `ARCHITECTURE.md` for this project's actual layer definitions.

## The Knowledge Base Layout

```
ARCHITECTURE.md              # Top-level architecture map (root level)
docs/
├── QUALITY_SCORE.md               # Quality grades per domain
├── GIT_WORKFLOW.md          # Branching, commits, PRs, merge strategy
├── INFRASTRUCTURE.md        # Services, CI/CD, cloud, databases
├── PLANS.md                 # Planning conventions
├── PRODUCT_SENSE.md         # Product context and domain language
├── RELIABILITY.md           # SLAs, error handling, observability
├── SECURITY.md              # Auth, data protection, secrets
├── design-docs/             # Design decisions with verification status
│   ├── index.md
│   └── core-beliefs.md
├── exec-plans/              # Execution plans as first-class artifacts
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/               # Auto-generated docs
├── product-specs/           # Product specifications
├── references/              # External reference material (llms.txt)
└── encyclopedia/            # Auto-generated codebase knowledge
```

Plans are first-class artifacts. Active plans, completed plans, and known technical debt
are all versioned and co-located, allowing agents to operate without relying on
external context.
